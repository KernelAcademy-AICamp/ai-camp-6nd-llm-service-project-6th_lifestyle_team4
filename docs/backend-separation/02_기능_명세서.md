# 기능 명세서 (Feature Specification)

> 서비스: **Daily Script — 오늘의 명대사** (iOS 앱명 *Curtaincall*) · 동일 Supabase 백엔드를 **Android·iOS·Web PWA** 3개 클라이언트가 공유.
> 본 문서는 분리될 `backend`가 책임지는 **기능 전체**와 클라이언트별 구현 현황을 정의한다. 백엔드 객체 상세는 [03_API_명세서.md](03_API_명세서.md) 참조.

---

## 1. 서비스 개요

영화·드라마·연극·뮤지컬·오페라·소설·시·수필 등 창작물의 **명대사/명문장을 카드 형태로 매일 제공**하고, 둘러보기·북마크·감상 공유(피드/하이라이트/댓글)·게이미피케이션(실타래 재화·출석)·카드 이미지 공유를 제공하는 라이프스타일 앱.

- **콘텐츠 생산**: 관리자가 `upload_web`(내부 도구)에서 대본/공개도서를 LLM으로 카드화 → 검수 큐 → 승격 → 운영 `cards` 노출 (§16).
- **콘텐츠 소비**: 일반 사용자가 3개 클라이언트로 카드를 소비·상호작용.
- **AI 고지**: 카드의 작품정보·키워드·장면설명·의의는 LLM 자동 생성물(약관 제7조).

## 2. 사용자·인증 모델

### 2.1 신원 모델 (3 클라이언트 공통)

- 모든 사용자별 데이터는 `users.user_id`(bigint)로 키잉되며, `users.anonymous_id = auth.uid()`(GoTrue UUID)로 인증과 매칭된다.
- 로그인 후 `ensure_user_row(p_nickname)` RPC로 `users` 행을 **원자적 get-or-create**.
- **프로바이더 간 계정 연결 없음**: 다른 인증수단 = 다른 `auth.uid()` = 다른 계정.
- **ID 로그인**: 사용자가 입력한 "아이디"를 합성 이메일(`<id>@user.local`, 비정규 ID는 FNV-1a 32bit 해시)로 변환. **이 매핑은 3 클라이언트가 바이트 단위로 동일**해야 계정이 공유됨.

### 2.2 인증 수단 — 클라이언트별

| 인증 수단 | Android | iOS | Web PWA | 비고 |
|---|---|---|---|---|
| 이메일/ID + 비밀번호 | ✅ | ✅ | ✅ | `signUp`/`signIn`; ID→합성이메일 |
| Google | ✅ 네이티브(Credential Manager, IDToken) | ✅ 웹리다이렉트(OAuth) | ✅ OAuth | Android만 브라우저 미경유 |
| Kakao | ✅ OAuth | ✅ OAuth | ✅ OAuth | 대시보드 Kakao provider 설정 필요 |
| Apple | ❌ | ✅ 코드완성(**대시보드 미설정→현재 비동작**) | ❌ | iOS 전용 |
| 익명(anonymous) | ❌ **폐지** | ❌ **폐지** | ❌ **폐지** | "유령 익명 유저" 정리로 전 클라이언트 비활성 |
| 비로그인 게스트(읽기전용) | ✅ (`userId=0L`) | ✅ (`userId=nil`) | ✅ (`userId=null`) | 쓰기 차단, 읽기만 |

### 2.3 권한 계층

- **게스트(비로그인)**: 공개 SELECT만(카드·작품·피드·하이라이트·댓글·공지 읽기). 쓰기 전부 차단.
- **회원(authenticated, 비익명)**: 자기 데이터 CRUD + 작성형 기능(댓글·피드·하이라이트·좋아요·북마크·실타래·출석·공유·신고/차단).
- **관리자(`app_metadata.role='admin'`, `@admin.local`)**: 콘텐츠 쓰기(works/cards/genres/notices/candidates) + Storage 쓰기. `upload_web`에서만 사용. service_role로만 부여 가능.

> RLS 규칙: 대부분의 작성형 테이블 INSERT는 **"본인 + 비익명 JWT"** 조건. 상세 §03.

## 5. 기능별 상세 명세

각 기능: **설명 / 사용자 스토리 / 동작 / 백엔드 객체 / 비즈니스 규칙·제약**.

### F1. 카드 · 콘텐츠

- **설명**: 명대사 카드(인용구 + 극본발췌 + 장면설명 + 키워드 + 의의 + 온도/강도 + 작품메타)를 제공.
- **사용자 스토리**: "오늘의 명대사 한 장을 받고, 아카이브에서 장르·검색으로 둘러보고, 상세에서 원문/번역을 본다."
- **동작**:
  - 카드 목록: `cards` + 임베드 `works(work_genres(genres))`, `card_id desc`, **1000건/페이지** `range()` 페이지네이션.
  - 단건: `cards` where `card_id`.
  - 조회수: RPC `increment_card_view(p_card_id)` (fire-and-forget).
  - 이중언어: `*_original` 컬럼(원문) ↔ 기본 컬럼(번역); 클라이언트가 언어 폴백.
  - 추천 가중: 클라이언트측 키워드→테마 분류기(`card-theme.js`/`CardTheme`/`Recommend`) + `users.pref_*`.
- **백엔드 객체**: `cards`, `works`, `genres`, `work_genres`; RPC `increment_card_view`.
- **규칙/제약**:
  - `cards.temperature/intensity` **CHECK 1–5**.
  - `works.format` enum **`work_format`** ∈ {movie,drama,play,musical,opera,novel,poem,essay,prose}. 범위 밖 insert 실패.
  - SELECT 공개(게스트 포함). 쓰기는 관리자만.

### F2. 북마크 · 라이브러리

- **사용자 스토리**: "마음에 든 카드를 저장하고, 라이브러리에서 모아본다."
- **동작**: `user_bookmarks` toggle(insert/delete), 본인 목록 조회(임베드 카드/작품), 카드별 카운트는 view `card_bookmark_counts`.
- **익명→회원 이관**: 로그인 시 이전 익명 user의 북마크를 새 user로 복사 후 옛 `users` 행 삭제(`UNIQUE(user_id,card_id)`로 dedup; iOS는 upsert ignoreDuplicates).
- **규칙**: `UNIQUE(user_id, card_id)`. 본인만 R/W(owner-only RLS).

### F3. 피드 — "오늘의 한줄"

- **사용자 스토리**: "카드에 대한 짧은 감상을 공개 피드에 남기고 본다."
- **동작**: `feed_posts` 목록(최신 50, 임베드 카드)·단건·내 글(100)·작성·수정·삭제.
- **규칙**: `body` **trim 1–300자**. INSERT=본인+비익명. UPDATE/DELETE=본인. SELECT 공개.

### F4. 하이라이트

- **사용자 스토리**: "본문에서 문장을 선택해 인용 카드로 만들고(노트 첨부) 공유·보관한다."
- **동작**: `card_highlights` 목록(50)·단건·내 것(100)·작성·삭제.
- **규칙**: `selected_text` **trim 1–2000자**, `user_note` NULL 또는 **≤500자**. INSERT=본인+비익명. SELECT 공개.

### F5. 댓글 · 대댓글 · 좋아요 (3개 표면)

세 콘텐츠 표면(카드 / 하이라이트 / 피드포스트) 각각에 동일 구조의 댓글+좋아요.

- **댓글 테이블**: `card_comments`, `card_highlight_comments`, `feed_post_comments` — 각 `parent_comment_id` 자기참조(대댓글), `body` **trim 1–500자**.
- **댓글 좋아요**: `comment_likes`, `card_highlight_comment_likes`, `feed_post_comment_likes` — PK `(comment_id,user_id)`.
- **콘텐츠 좋아요(피드/하이라이트 본문)**: `content_likes`(PK `(user_id,target_type,target_id)`, `target_type` ∈ {feed_post,highlight}) + view `content_like_counts`; 토글은 RPC `toggle_content_like(p_user_id,p_target_type,p_target_id)` → `{liked,count}`.
- **카드 댓글 수**: `cards.comment_count`를 트리거 `sync_card_comment_count`가 동기화.
- **규칙**: INSERT=본인+비익명. UPDATE/DELETE=본인. SELECT 공개.

### F6. 실타래(yarn) 경제

- **설명**: 가상 재화. 잔액은 **서버 `users.yarn_balance`만**이 정본(열람 게이트·매일 무료분 폐지).
- **획득**: 첫 열람 +300(`reward_yarn_first_view`, `yarn_card_rewards`로 `(user_id,card_id)` dedup), 출석 +100(`check_in_attendance`), 추천인 +600(`redeem_referral`).
- **소비**: `spend_yarn(p_amount)`(공유 테마 잠금해제 등), `consume_yarn()`(−1), QA용 `grant_yarn(p_n)`.
- **백엔드**: 모든 변동은 `SECURITY DEFINER` RPC를 통해 원자적으로. 반환값 음수=오류(−1 인자없음/잔액부족 등).
- **규칙/주의**: `grant_yarn`은 무제한 발행 가능(현재 결제 없음→의식적 보류; 결제 도입 시 차단 필요). iOS는 스토어 심사를 위해 유료 구매 티어를 제거(earn-only).

### F7. 출석

- **사용자 스토리**: "매일 출석하고 실타래 보상을 받는다."
- **동작**: RPC `check_in_attendance(p_reward=100)` — **KST 기준 당일 첫 출석**이면 `attendance` 행 생성 + `yarn_balance += p_reward`, 반환 `{rewarded,balance,today}`. 본인 이력은 `attendance` SELECT(owner-only).
- **규칙**: PK `(user_id, attended_date)`로 1일 1회. 익명 차단. 쓰기는 RPC로만(직접 INSERT 정책 없음).

### F8. 공유 카드 (이미지 공유)

- **사용자 스토리**: "카드를 예쁜 배경(무료/프리미엄/로열)에 얹어 이미지로 저장·카카오톡/공유한다."
- **동작**:
  - 배경 목록: `share_backgrounds`(active만, tier premium/royal, `image_url`은 Storage `share-backgrounds` 공개 URL).
  - 테마 잠금해제: `purchase_share_theme(p_theme_id,p_price)`(실타래 차감+`share_theme_unlocks` insert), 소유목록 `share_theme_unlocks` SELECT. (구 OZ테마: `purchase_oz_theme`+`oz_theme_unlocks`.)
  - 공유 링크: `create_share_link(p_referrer_id,p_card_id,p_bg_id,p_quote_b64)` → 6자 short_id(`share_links`), `/m/?s=<id>` 로 소비.
  - 공유수: `increment_share_count(p_card_id)`.
- **클라이언트 차이**: Android/Web=DB 배경 전체. iOS=DB 배경/유료티어 미구현(로컬 공유만), 단 `increment_share_count`는 호출.
- **규칙**: `share_backgrounds.tier` **CHECK ∈ {premium,royal}**. 무료 배경은 클라이언트 자산. Storage 쓰기는 관리자만, 읽기는 공개.

### F9. 알림 (확성기) — Android/Web only

- **사용자 스토리**: "내 글/하이라이트에 댓글·대댓글이 달리면 알림으로 받는다."
- **동작**: 댓글 INSERT 시 트리거(`notify_on_feed_post_comment`/`notify_on_highlight_comment`)가 `notifications` 행 생성. 클라이언트가 `recipient_user_id`로 목록/안읽음 수 조회, `is_read` 일괄 갱신. (+헤더 캠페인 버튼·배지, 포그라운드 폴링.)
- **규칙**: `kind` ∈ {post_comment, comment_reply, highlight_comment, highlight_comment_reply}. RLS 관대(`using(true)`)이나 INSERT는 DEFINER 트리거에서만 발생, 클라이언트는 `recipient_user_id` 필터로 자기 것만 조회. **iOS 미구현**(백엔드 계약은 존재).
- **누락 표면**: 카드 댓글(`card_comments`)에는 알림 트리거 없음(피드/하이라이트 댓글만).

### F10. 공지사항

- **동작**: `notices`(published만, `pinned desc, created_at desc`, 100). 이미지 첨부는 Storage `notice-images`(공개 읽기, 관리자 쓰기).
- **규칙**: `tag` ∈ {update,notice,event}. SELECT는 published 또는 관리자. 쓰기 관리자만.

### F11. 신고 / 차단 (UGC moderation) — iOS only (현재)

- **설명**: App Store 1.2 UGC 정책 대응.
- **동작**:
  - 신고: `report_content(p_content_type,p_content_id,p_reason)` → `content_reports`(본인 dedup).
  - 차단: `block_user(p_blocked_user_id)`/`unblock_user(...)` → `user_blocks`. 차단목록 `user_blocks` SELECT로 클라이언트 필터.
- **규칙**: `content_type` ∈ {feed_post,card_comment,highlight,highlight_comment,feed_post_comment}. `reason` 1–50자. UNIQUE `(reporter,content_type,content_id)`. 회원만(익명 차단). `content_reports`는 RLS 정책 無→직접접근 차단, RPC로만 기록.
- **분리 권고**: Android/Web도 스토어/정책상 동일 기능 필요 가능 → 백엔드 계약 유지.

### F12. 추천 / 선호도 / 온보딩

- **동작**: 온보딩에서 장르/테마 선호 수집 → `users.pref_genres`(jsonb)/`pref_themes`(jsonb)/`pref_any`(bool) 업데이트. 추천 점수는 클라이언트측 분류기(`card-theme`/`Recommend`)가 카드 키워드와 선호를 매칭. Web은 주로 localStorage, 서버 동기화는 update.
- **규칙**: 본인만 update.

### F13. 계정 / 프로필 / 탈퇴

- **동작**:
  - 가입 보장: `ensure_user_row(p_nickname)`.
  - 가용성: `email_available(p_email)`(가입), `find_email_by_login_id(p_login_id)`(로그인/재설정).
  - 프로필: `users.nickname/gender/age_group/login_id` update.
  - 탈퇴: **두 경로 공존** — Android/iOS=`delete_account()`(no-arg, auth.uid() 기준), Web=`delete_my_account(p_user_id)`. 둘 다 `public.users`+`auth.users`+종속 콘텐츠 삭제(FK CASCADE).
- **규칙**: `gender` ∈ {male,female,other}, `age_group` ∈ {10s…90s}. `login_id`는 lower() UNIQUE. 본인만.

### F14. 추천인 (레퍼럴)

- **동작**: 공유 링크에 `referrer_id` 포함 → 신규 가입 시 Web에서 `redeem_referral(p_referrer_id,p_referee_id)` → 양측 +600 실타래(`referrals`로 referee dedup).
- **규칙**: referee 1회. 반환 음수=오류. **Web에서만 정산**(Android/iOS는 링크만 발급).

### F15. 피드백

- **동작**: 클라이언트 폼 → (Web) `POST /api/feedback` 프록시 → Google Apps Script. Android/iOS는 Apps Script 직접 POST. 필드 `rating,gender,age,liked,improve,message,email,page`.
- **규칙**: Supabase 미경유. `page`로 출처 구분(user/android/ios). 본문 길이 캡·64KB 제한(서버리스).

### F16. 분석

- **동작**: Amplitude(세션·이벤트·로그인 시 user_id 식별) + Microsoft Clarity(세션 리플레이). 키는 공개값, 미설정 시 no-op.
- **클라이언트**: Android·Web 구현, **iOS 미구현**.

### F17. 관리자 콘텐츠 파이프라인 (upload_web)

§16 별도 상세. 일반 사용자 비대상.

## 6. 관리자(콘텐츠) 파이프라인 상세 — upload_web

> 내부 운영 도구. 관리자(`@admin.local`, `app_metadata.role='admin'`)만. DB 쓰기는 **service_role**. 모든 라우트 `requireAdmin` 가드. 엔드포인트 I/O는 [03_API_명세서.md](03_API_명세서.md) §6.2.

**파이프라인 흐름:**

```
[소스 확보]
  ├─ 파일 업로드(PDF/DOCX/HWP/TXT ≤50MB)            → /api/extract (multipart)
  ├─ 외부 공개도서: Gutenberg/Gutendex               → /api/fetch-source, /api/gutenberg-list
  └─ 한국 공개문학: ko.wikisource                     → /api/fetch-source
        │
        ▼
[LLM 카드 추출]  /api/extract  (Anthropic Claude)
  - 카테고리: screen|opera|play|novel|poem|essay|prose
  - 명대사 인용구 검증(verbatim) + 작품/카드 JSON
  - Wikiquote/Wikipedia/namu.wiki 인용 시드 보강
        │
        ▼
[검수 큐 적재]  /api/save  → card_candidates (status=pending)
  - works insert, genres upsert + work_genres 링크
  - 작가명 한글화(Anthropic), 카드 정규화/검증
        │
        ▼
[관리자 검수]  /api/candidates  (GET 목록 / POST claim·release·decide·save·delete)
  - claim: 10분 TTL 잠금
  - decide(approved) → RPC promote_candidate → cards (운영 노출)
        │
        ▼
[보강 백필]
  ├─ /api/backfill-characters  (works.characters)
  ├─ /api/backfill-intro       (works.intro)
  └─ 번역: /api/translate-field, /api/translate-fields, /api/translate-card-batch (EN↔KO)
  └─ /api/classify-keywords    (키워드 분류)
```

**관련 테이블**: `card_candidates`(검수 큐), `works`/`cards`/`genres`/`work_genres`(운영), `gutenberg_books`/`gutenberg_book_text`/`gutendex_cache`(공개도서 캐시).
**관련 RPC**: `promote_candidate`, `search_gutenberg_by_topic`, `search_gutenberg_by_title`.

## 7. 비기능 요구 (분리 후 백엔드가 보장)

- **인증/인가**: GoTrue JWT + Postgres RLS 2중 방어. service_role은 서버에서만.
- **데이터 정합성**: 재화/출석/좋아요/신고 등 동시성 민감 작업은 `SECURITY DEFINER` RPC로 원자화.
- **이중언어**: 원문/번역 컬럼 병행, 부분 인덱스로 원문보유 카드 탐색.
- **실시간(부분)**: `supabase_realtime` 퍼블리케이션에 cards·works·user_bookmarks·card_comments·comment_likes·feed_posts·card_highlights·notices 등록(클라이언트 현행은 폴링 위주, 계약상 가용).
- **콘텐츠 신뢰성**: 카드 메타는 AI 생성물(부정확 가능, 약관 고지).
- **국외 처리**: Supabase/Amplitude/Clarity 해외 리전(개인정보 위탁 고지).
