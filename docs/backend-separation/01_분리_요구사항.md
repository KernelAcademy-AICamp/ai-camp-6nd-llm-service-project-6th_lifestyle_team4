# 백엔드 분리 — 필요사항 정의서 (What's Needed)

> 본 문서는 **코드를 수정하지 않고**, 현재 구조를 분석해 `android_frontend / ios_frontend / backend` 분리에 **무엇이 필요한지**를 정의한다.
> 동반 문서: [02_기능_명세서.md](02_기능_명세서.md), [03_API_명세서.md](03_API_명세서.md)

---

## 0. 한 줄 요약

이 프로젝트의 "백엔드"는 **별도의 API 서버가 아니라 Supabase(BaaS) 프로젝트 그 자체**다.
세 클라이언트(Android·iOS·Web PWA)는 각자 **Supabase anon 키로 Postgres(PostgREST)·RPC·Storage·Auth를 직접 호출**한다.
따라서 "백엔드 분리"의 실체는 *서버 코드를 떼어내는 것*이 아니라 다음 세 가지다.

1. **흩어진 백엔드 자산(SQL 스키마·RLS·RPC·서버리스 함수)을 단일 `backend/`로 통합**한다.
2. 그 위에 **안정적인 API 계약**(본 산출물 02·03 문서)을 고정한다.
3. 각 프론트엔드는 **그 계약만 소비**하도록 의존 방향을 정리한다(프론트가 백엔드 SQL을 보유하는 현재 안티패턴 제거).

```
            ┌─────────────────────── Supabase 프로젝트(백엔드) ───────────────────────┐
            │  Postgres + PostgREST(REST) │ RPC(함수) │ Storage(버킷) │ GoTrue(Auth)  │
            └───────────────▲────────────────────▲────────────────────▲──────────────┘

        ┌───────────────────┼────────────────────┼────────────────────┼───────────────┐
        │                   │                    │                    │               │
   ┌────┴────┐        ┌─────┴─────┐        ┌─────┴──────┐       ┌─────┴─────┐   ┌─────┴──────┐
   │ Android │        │   iOS     │        │  Web PWA   │       │  Web PWA  │   │ upload_web │
   │ (kt)    │        │ (swift)   │        │  브라우저   │       │ /api/*    │   │ /api/*     │
   └─────────┘        └───────────┘        └────────────┘       └───────────┘   └─────┬──────┘
   직접호출            직접호출              직접호출              (config/feedback)    │ service_role
                                                                                      │ + Anthropic
                                                                              관리자 전용 LLM 파이프라인
```

**핵심:** 비즈니스 로직 대부분이 **Postgres 안(RLS 정책 + `SECURITY DEFINER` RPC 함수)**에 있다. 실타래(yarn) 경제, 출석, 좋아요 토글, 신고/차단, 계정 삭제, 공유링크 발급 등은 전부 DB 함수다. "백엔드 코드"라고 부를 수 있는 별도 서버 레이어는 **Vercel 서버리스 함수 14개**(소비자 BFF 2 + 관리자 12)가 전부이며, 그나마 대부분은 관리자(콘텐츠 적재) 전용이다.

### 1.3 진단된 문제 (분리해야 하는 이유)

| # | 문제 | 근거 |
|---|---|---|
| P1 | **백엔드 자산이 프론트 안에 흩어짐** | SQL 스키마가 `upload_web/supabase/migrations`(파일 57개) **와** `main_app/android/supabase`(14개) 두 곳에 분산. iOS는 자기 SQL이 없어 Android 폴더에 의존. |
| P2 | **마이그레이션 번호 충돌·중복** | `017_`×2, `042_`×2, `005/008/009/010/045_`×2 등 번호 중복(파일 57개·번호 047까지). Android `10~12`는 web `042/045/046`와 기능상 동일한 재실행본(주석·헤더가 달라 바이트는 불일치; 특히 android 12는 `purchase_share_theme`에 방어구문 추가로 본문도 다름). 단일 정렬 순서가 없음. |

| P4 | **클라이언트 기능 분기로 백엔드 계약이 암묵화됨** | iOS에만 신고/차단(`report_content`/`block_user`)·첫열람보상(`reward_yarn_first_view`)이 있고, Android에만 알림(`notifications`)·공유배경 DB(`share_backgrounds`)가 있음. 어느 클라이언트도 백엔드 전체 계약을 알지 못함. |
| P5 | **마이그레이션 거버넌스 부재(수동 실행)** | 메모리·README상 "SQL Editor에서 수동 1회 실행" 다수. 누가 무엇을 언제 적용했는지 추적 불가, 신규 프론트 개발자가 스키마를 재현할 방법이 없음. |
| P6 | **단일 환경(프로덕션 직접 작업)** | dev/staging Supabase 프로젝트 분리가 없음(추정). 프론트 2팀이 같은 DB를 건드릴 위험. |

## 2. 목표 구조

### 2.1 제안 디렉터리

```
repo-root/
├── backend/                       # ★ 단일 백엔드 출처 (Supabase + 서버리스)
│   ├── supabase/
│   │   ├── migrations/            # 통합·재번호화된 마이그레이션 (단일 순서)
│   │   ├── functions/             # (선택) Supabase Edge Functions로 이전 시
│   │   ├── policies/              # RLS 정책 모음(가독성용, 마이그레이션과 동기화)
│   │   ├── seed/                  # 샘플·초기 데이터
│   │   └── config.toml            # supabase CLI 프로젝트 설정
│   ├── serverless/                # Vercel 함수(현행 유지 시) — admin/LLM 파이프라인
│   │   ├── api/                   # extract, translate-*, save, candidates, ...
│   │   └── lib/                   # anthropic.js, supabase-admin.js, auth.js, prompts.js
│   ├── docs/                      # ← 본 분리 문서 3종을 이 위치로 이동 가능
│   │   ├── api-spec.md
│   │   └── feature-spec.md
│   └── README.md                  # 스키마 적용·배포·환경변수 안내(현 README들 통합)
│
├── android_frontend/              # = 기존 main_app/android (supabase/ 제거)
├── ios_frontend/                  # = 기존 main_app/ios
└── web_frontend/                  # = 기존 web_pwa  (※ 4번째 클라이언트, 2.3 참조)
```

### 2.2 분리 원칙

- **단일 진실 출처(SSOT):** 스키마·RLS·RPC·Storage 정책은 오직 `backend/supabase/`에만 존재. 프론트는 SQL을 한 줄도 보유하지 않는다.
- **계약 우선(Contract-first):** 프론트와 백엔드의 유일한 접점은 [03_API_명세서.md](03_API_명세서.md)에 고정된 **테이블 스키마 + RPC 시그니처 + RLS 규칙 + Storage 버킷**이다. 백엔드는 이 계약의 **합집합**(모든 클라이언트가 쓰는 모든 객체)을 책임진다.
- **느슨한 결합:** 프론트는 Supabase URL·anon 키와 API 계약만 알면 빌드 가능해야 한다(백엔드 내부 구현 비의존).

### 2.3 ⚠️ 의사결정 필요 — web_pwa / upload_web 의 위치

사용자가 지정한 3폴더(`android_frontend / ios_frontend / backend`)에는 **현재 존재하는 두 웹 앱이 빠져 있다.** 다음 권고로 진행한다(차단 사항 아님, 추후 조정 가능):

| 대상 | 권고 | 근거 |
|---|---|---|
| **web_pwa** | **`web_frontend/`로 승격**(4번째 프론트엔드) | Android/iOS와 동일하게 anon 키로 백엔드를 직접 호출하는 **소비자 클라이언트**다. 프론트엔드 카테고리에 속한다. (단, `/api/config`·`/api/feedback` 두 서버리스는 BFF 성격이라 web_frontend에 동봉하거나 backend/serverless로 이전 — 아래 3.3) |
| **upload_web** | **`backend/` 산하 admin 도구**로 편입(또는 `backend/admin/`) | 일반 사용자 비대상 **내부 운영 도구**다. `service_role` 키로 DB에 직접 쓰고, Anthropic LLM으로 카드를 생성·번역·검수해 적재한다. 사실상 **백엔드의 콘텐츠 적재 파이프라인**이므로 백엔드와 함께 두는 것이 응집도가 높다. |

> 만약 "백엔드 = 순수 Supabase + 마이그레이션만"으로 좁히고 싶다면, `upload_web`을 `admin_tooling/`으로 별도 분리하고 `web_pwa`를 `web_frontend/`로 두는 4-패키지 구성도 가능하다. 어느 쪽이든 **SQL 마이그레이션은 backend로 통합**한다는 원칙은 동일하다.

## 3. 필요한 작업 (What's Needed) — 항목별

### 3.1 [필수] 디렉터리 재배치

- `main_app/android` → `android_frontend`
- `main_app/ios` → `ios_frontend`
- `web_pwa` → `web_frontend` (권고)
- `backend/` 신설, 아래 자산 이동.
- 경로 변경에 따라 깨질 수 있는 참조 점검(코드 수정은 본 작업 범위 밖이나 사전 식별 필요):
  - iOS `.xcodeproj` 내부 경로, Android `settings.gradle`/`local.properties` 위치, Vercel 프로젝트의 **Root Directory** 설정, `.gitignore`의 `main_app/ios/...` 경로(L22).

### 3.2 [필수] SQL 마이그레이션 통합 (P1·P2 해소)

- `upload_web/supabase/migrations/*`(파일 57개·번호 047까지) + `main_app/android/supabase/*`(14)를 **`backend/supabase/migrations/`로 병합**.
- **번호 충돌 정리 규칙:**
  - Android `10/11/12`는 web `042_share_backgrounds / 045_attendance / 046_share_theme_unlocks`와 **기능상 동일**(주석·헤더만 다름)한 재실행본 → 중복 제거(web 본을 정본으로). ⚠️ 단 android `12`의 `purchase_share_theme`에는 web 046에 없는 방어구문(`order by user_id limit 1`)이 추가돼 본문이 다르므로, **정본 채택 시 이 가드를 보존**할 것.
  - Android `01~09, 13, 14`(RLS·comment_count·yarn·feed_post_comments·highlight_comments·comment_likes·익명정리·중복유저정리)는 web에 없는 **고유 객체** → 통합 순서 뒤에 재번호 부여.
  - 함수 다중정의는 **최종본만 채택**: `promote_candidate`=026, `reward_yarn_first_view`=038(+300), `delete_my_account`=047.
- **타임스탬프 기반 재번호화 권장**(`supabase migration new`의 `YYYYMMDDHHMMSS_` 형식)으로 영구적 순서 보장.
- ⚠️ **검증 필요(중대):** `users / user_bookmarks / user_daily_cards / user_card_selections / user_preferences` 5개 **기반 테이블은 어떤 SQL 파일에도 `CREATE TABLE`이 없다**(마이그레이션 이전에 대시보드에서 수동 생성된 것으로 추정). 분리 시 **라이브 DB에서 정확한 DDL을 추출(`pg_dump --schema-only`)해 베이스 마이그레이션으로 박제**해야 신규 환경 재현이 가능하다. (상세는 03 문서 §부록)

### 3.3 [필수] 서버리스 함수 정리 (P 보조)

현재 서버리스(=실제 커스텀 백엔드 코드)는 14개(소비자 BFF 2 + 관리자 12):

| 위치 | 함수 | 성격 | 키 | 분리 후 위치 권고 |
|---|---|---|---|---|
| `web_pwa/api/config.js` | anon 설정 노출 | 소비자 BFF | (env만 노출) | web_frontend 동봉 또는 backend/serverless |
| `web_pwa/api/feedback.js` | 피드백 프록시(→Google Apps Script) | 소비자 BFF | 없음 | web_frontend 동봉 또는 backend/serverless |
| `upload_web/api/*` (12개) | LLM 추출·번역·검수·Gutenberg·config | **관리자/백엔드** | **service_role + Anthropic** | **backend/serverless** |
| `upload_web/lib/*` | anthropic·supabase-admin·auth·prompts | 서버 전용 | service_role | **backend/serverless/lib** |

- ⚠️ **Vercel Hobby 함수 12개 제한**: `upload_web`은 현재 12개로 **이미 한계에 도달**. backend로 합치면서 web의 2개까지 더하면 **14개 → Hobby 초과**. 따라서 **소비자 BFF(config/feedback)는 web_frontend에 두고, admin 12개만 backend에 둔다**는 분리가 운영상 안전하다(그래도 12개로 한계라 Pro 승급 또는 함수 통합 검토 필요).
- (선택) `upload_web/api`의 Anthropic 호출을 **Supabase Edge Functions**(Deno)로 이전하면 Vercel 의존을 끊고 백엔드를 완전히 Supabase로 응집할 수 있으나, `pdf-parse`/`mammoth`/`sharp` 등 Node 전용 의존성 이식 비용이 큼 → **현행 Vercel 유지를 권고**.

### 3.4 [필수] 설정·시크릿 표준화 (P3 해소)

분리의 핵심 산출물 중 하나. 각 프론트가 **동일한 계약(Supabase URL + anon 키)**을 **환경별로** 주입받도록 통일한다.

| 클라이언트 | 현재 | 목표 |
|---|---|---|
| Android | `local.properties`→`BuildConfig` (git-ignored) | 유지(이미 양호). dev/prod flavor로 키 분리. |
| iOS | `Config.swift`·위젯에 **하드코딩**(키가 git에 노출) | `.xcconfig`(git-ignore) 또는 Info.plist 빌드설정 주입으로 전환. **anon 키 git 이력 제거 검토.** |
| Web | `/api/config`(env 주입) | 유지(이미 양호). |

- 공통 문서화: `SUPABASE_URL`, `SUPABASE_ANON_KEY`(공개), 그리고 **서버 전용** `SUPABASE_SERVICE_ROLE_KEY`·`ANTHROPIC_API_KEY`·`ANTHROPIC_MODEL`(기본 `claude-sonnet-4-6`)·`FEEDBACK_ENDPOINT`·`AMPLITUDE_API_KEY`·`CLARITY_PROJECT_ID`·`GOOGLE_WEB_CLIENT_ID`.
- 원칙: **anon 키 = 공개값(클라이언트 OK)**, **service_role = 절대 클라이언트/저장소 노출 금지**(현재도 `lib/supabase-admin.js`가 `/api`에서만 import — 유지).

### 3.5 [필수] API 계약 문서화 — 본 작업 산출물

- [02_기능_명세서.md](02_기능_명세서.md): 기능 단위 명세 + 클라이언트별 커버리지 매트릭스.
- [03_API_명세서.md](03_API_명세서.md): 테이블·컬럼·타입, RPC 시그니처(정확), RLS 접근 규칙, Storage 버킷, 서버리스 엔드포인트 I/O, 오류 코드.

### 3.6 [권장] 기능 분기 정합화 (P4 해소)

백엔드는 **모든 클라이언트 기능의 합집합**을 이미 지원한다(스키마에 다 있음). 분리 후 권장:

- iOS에 누락된 **알림(`notifications`)·공유배경(`share_backgrounds`/Storage)** 백엔드 계약 명시 → iOS팀이 구현 여부 결정.
- Android에 누락된 **신고/차단(`report_content`/`block_user`/`unblock_user`)** 계약 명시 → 스토어 UGC 정책상 Android도 필요할 수 있음.
- 백엔드 입장에서는 **계약을 줄이지 말 것**(한 클라이언트만 써도 유지).

### 3.7 [권장] 환경·거버넌스 (P5·P6 해소)

- **Supabase CLI 도입**: `supabase/config.toml` + `supabase db push`/`migration` 워크플로로 수동 SQL 실행을 대체.
- **환경 분리**: `dev` / `prod` 2개 Supabase 프로젝트. 프론트 빌드 설정으로 타겟 전환.
- **CI**: PR 시 마이그레이션 lint + (가능하면) 임시 DB에 적용 검증.
